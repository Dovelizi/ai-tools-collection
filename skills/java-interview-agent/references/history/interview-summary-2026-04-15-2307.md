### 1.1 Java 基础与 JVM

**Q1: ConcurrentHashMap 在 JDK 7 和 JDK 8 中线程安全的实现机制分别是什么？JDK 8 为什么要放弃分段锁？**

📝 **参考答案（完整版）**：

**JDK 7 — Segment 分段锁：**

- 数据结构：`Segment[]` 数组，每个 Segment 继承 `ReentrantLock`，内部维护一个 `HashEntry[]` 数组
- 默认 16 个 Segment，即最多 16 个线程并发写
- 锁粒度：一个 Segment（一组桶），不是单个桶
- 读操作不加锁（volatile 保证可见性），写操作锁单个 Segment

**JDK 8 — CAS + synchronized：**

- 数据结构：与 HashMap 一致，`Node[]` 数组 + 链表/红黑树
- 锁粒度：**单个桶的头节点**（synchronized 锁住链表头），并发度 = 数组长度
- 无竞争时用 **CAS** 写入（如空桶直接 CAS 放入）
- 有竞争时 **synchronized** 锁住链表头节点，只锁一个桶

**JDK 8 放弃分段锁的三个原因：**

1. **锁粒度更细**：Segment 锁一组桶（并发度=16），synchronized 锁单个桶（并发度=数组长度），并发性能大幅提升
2. **内存开销更小**：Segment 继承 ReentrantLock + 内部维护 HashEntry 数组，结构嵌套深、内存占用大
3. **synchronized 优化**：JDK 6+ 引入锁升级（无锁 → 偏向锁 → 轻量级锁 → 重量级锁），低竞争下性能不输 ReentrantLock

**synchronized 锁升级顺序**：无锁 → 偏向锁 → 轻量级锁 → 重量级锁（单向升级不可降级）

👤 **候选人回答**：知道 JDK 7 用 Segment 分段锁 + ReentrantLock，JDK 8 用 CAS + synchronized。提到了锁粒度差异和 synchronized 锁升级优化。

📊 **评价**：⚠️ 核心思路正确，但锁粒度描述不够精确（说"JDK 7 锁一个桶"，实际是锁一组桶/一个 Segment）；锁升级顺序有些混乱（偏向锁和轻量级锁说反了两次）

💡 **面试官补充讲解**：纠正了 Segment 锁的是"一组桶"而非"单个桶"；补充了放弃分段锁的三个原因（粒度、内存、synchronized 优化）；明确锁升级正确顺序

**Q2（追问）: ConcurrentHashMap 的 `size()` 方法在 JDK 7 和 JDK 8 中分别是怎么实现的？**

📝 **参考答案（完整版）**：

**JDK 7 — 先乐观后悲观：**

1. 先不加锁，连续统计最多 **3 次**，每次累加所有 Segment 的 `count`
2. 每次统计同时记录 `modCount`（修改次数），如果相邻两次 modCount 总和一致，说明无并发修改，直接返回
3. 如果 3 次都不一致，退化为**全部 Segment 加锁**后再统计

设计思想：乐观锁 → 悲观锁的两阶段策略

**JDK 8 — 借鉴 LongAdder 的分散计数：**

```java
private transient volatile long baseCount;
private transient volatile CounterCell[] counterCells;
```

1. **无竞争时**：CAS 更新 `baseCount`
2. **CAS 竞争失败时**：将增量分散到 `CounterCell[]` 数组的不同槽位（通过线程 hash 映射）
3. **`size()` 调用时**：`baseCount + Σ(所有 CounterCell 的 value)`

核心思想：**热点分散**——多线程写不同 cell，最后汇总求和，与 `LongAdder` 原理一致。

👤 **候选人回答**：不了解

📊 **评价**：❌ 未掌握。需要重点学习 LongAdder 的热点分散思想。

💡 **面试官补充讲解**：详细讲解了 JDK 7 的乐观重试+悲观加锁策略，以及 JDK 8 的 baseCount + CounterCell 分散计数机制。建议学习 LongAdder 源码。

**Q3: JVM 常见垃圾收集器？G1 和 CMS 的核心区别？什么场景选 G1？**

📝 **参考答案（完整版）**：

**常见垃圾收集器（7 种）：**

| 收集器 | 作用区域 | 算法 | 特点 |
|--------|---------|------|------|
| Serial | 新生代 | 标记-复制 | 单线程，Client 模式默认 |
| ParNew | 新生代 | 标记-复制 | Serial 多线程版，搭配 CMS |
| Parallel Scavenge | 新生代 | 标记-复制 | 吞吐量优先 |
| Serial Old | 老年代 | 标记-整理 | 单线程 |
| CMS | 老年代 | **标记-清除** | 低延迟，但有内存碎片 |
| **G1** | 全堆（Region） | **标记-复制**（Region 间） | 可预测停顿，无碎片 |
| ZGC | 全堆 | 染色指针+读屏障 | 超低延迟（<1ms） |

**G1 vs CMS 核心区别：**

| 维度 | CMS | G1 |
|------|-----|-----|
| 内存模型 | **传统分代**（Young + Old 固定区域） | **Region 化**（2048 个 Region，动态分配 Eden/Survivor/Old/Humongous） |
| 回收算法 | **标记-清除** → 有内存碎片 | **标记-复制**（跨 Region 拷贝存活对象）→ 无碎片 |
| 停顿预测 | 不可控 | **可预测**（`-XX:MaxGCPauseMillis` 设目标停顿，G1 优先回收收益最大的 Region） |
| 适用堆大小 | 中小堆（< 6G） | **大堆**（6G 以上优势明显） |
| 新生代回收 | 需搭配 ParNew | **自己搞定**（Young GC + Mixed GC） |
| Full GC | 并发失败时 Serial Old 兜底 | Mixed GC 跟不上时退化为 Full GC |

**选 G1 的场景**：大堆内存（6G+）+ 对延迟敏感（需要可预测的停顿时间）

👤 **候选人回答**：提到 G1、CMS、ZGC；知道 G1 可预测停顿、Region 化（2048 个）、G1 无碎片。但 CMS 算法说成"标记复制"（应为标记清除），G1 适用场景说成"内存较小"（应为大堆）。

📊 **评价**：⚠️ 核心差异点答出了大部分，但两个关键细节说反了：CMS 算法名称和 G1 适用堆大小

💡 **面试官补充讲解**：纠正 CMS 是标记-清除（所以有碎片），G1 适合大堆（6G+）；补充了 7 种收集器对照表

### 1.2 Spring 框架

**Q4: Spring Bean 的完整生命周期？从 getBean() 到可以使用经历了哪些步骤？**

📝 **参考答案（完整版）**：

**Spring Bean 完整生命周期（7 个关键步骤）：**

```
1. BeanDefinition 加载 — 解析 @Component/@Bean/XML，注册到 BeanDefinitionMap
        ↓
2. 实例化（Instantiation）— 反射调用构造器 new 出对象（此时是"空壳"）
        ↓
3. 属性填充（Populate）— @Autowired/@Value 依赖注入（循环依赖在此触发三级缓存）
        ↓
4. Aware 回调 — BeanNameAware → BeanFactoryAware → ApplicationContextAware
        ↓
5. 初始化（Initialization）
   - BeanPostProcessor.postProcessBeforeInitialization()  ← 前置处理
   - @PostConstruct / InitializingBean.afterPropertiesSet() / init-method
   - BeanPostProcessor.postProcessAfterInitialization()  ← 后置处理（AOP 代理在此创建）
        ↓
6. 使用 — Bean 放入一级缓存（singletonObjects），可被 getBean() 获取
        ↓
7. 销毁 — @PreDestroy / DisposableBean.destroy() / destroy-method
```

**核心要点：**
- **实例化 ≠ 初始化**：实例化 = new 对象（空壳），初始化 = 执行 init 逻辑（填充完属性后）
- **实例化在前，初始化在后**
- BeanPostProcessor 包裹在初始化前后，AOP 代理在后置处理器中创建

**记忆口诀**：定义 → 实例化 → 填充 → Aware → 初始化（前置 → init → 后置）→ 使用 → 销毁

👤 **候选人回答**：提到了定义、初始化、前置后置拦截器、属性填充、实例化等环节，但**顺序错误**——说"先初始化再实例化"

📊 **评价**：⚠️ 知道有哪些环节，但实例化和初始化的顺序搞反了，这在面试中是明显扣分项

💡 **面试官补充讲解**：纠正顺序（实例化在前、初始化在后），补充完整 7 步生命周期和记忆口诀

**Q5（追问）: Spring 如何实现 `@Autowired Map<String, Interface>` 按名称注入多个实现类？底层机制是什么？**

📝 **参考答案（完整版）**：

核心处理类：`DefaultListableBeanFactory.doResolveDependency()`

处理流程：
1. **类型识别**：`AutowiredAnnotationBeanPostProcessor` 处理 `@Autowired`，发现目标类型是 `Map<String, T>`
2. **泛型解析**：通过 `ResolvableType` 解析出 Value 的泛型类型
3. **候选查找**：调用 `BeanFactory.getBeansOfType(T.class)` 从容器中找出所有实现类
4. **组装 Map**：Key = Bean 名称（`@Component("xxx")` 指定），Value = Bean 实例
5. **注入**：将 Map 注入到字段中

```java
@Autowired
private Map<String, OrderStatusAdapter> adapterMap;
// Spring 自动注入：{"supplier_A": SupplierAAdapter, "supplier_B": SupplierBAdapter, ...}
```

👤 **候选人回答**：不了解

### 1.3 中间件与数据存储

**Q6: Redis 分布式锁怎么实现？SETNX 加锁有哪些常见的坑？如何保证锁的可靠性？**

📝 **参考答案（完整版）**：

**标准实现：**
```bash
SET lock_key <UUID> NX EX 30
```
- `NX`：只在 key 不存在时设置（互斥）
- `EX 30`：设置 30 秒过期（防止宕机死锁）
- `<UUID>`：Value 用唯一标识（防止误删别人的锁）

**SETNX 常见的 5 个坑：**

| 坑 | 说明 | 解决方案 |
|---|---|---|
| SETNX 和 EXPIRE 分两步 | 中间宕机导致死锁 | `SET key value NX EX` 一条命令 |
| 锁过期但业务没执行完 | 其他线程获取锁，并发问题 | Redisson watchdog 看门狗续期（每 10 秒检查，续到 30 秒） |
| 误删别人的锁 | A 的锁过期，B 加锁成功，A 执行完删了 B 的锁 | Value 用 UUID，Lua 脚本先判断再删除 |
| Redis 主从切换丢锁 | 主节点加锁后宕机，从节点未同步 | RedLock（N/2+1 个节点加锁成功） |
| 不可重入 | 同一线程重复加锁死锁 | Redisson 用 hash 结构记录重入次数 |

**Lua 脚本原子删除：**
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

👤 **候选人回答**：
- `SET key value NX EX 30` 原子加锁 ✅
- Value 防止误删 ✅
- 看门狗续期机制（每 10 秒检查）✅
- RedLock 多数节点加锁 ✅
- Lua 脚本原子删除（先判断 Value 再删除）✅

📊 **评价**：✅ 回答非常全面，覆盖了分布式锁的几乎所有核心知识点，是今天表现最好的一题

💡 **面试官补充讲解**：补充了 SETNX 的 5 个坑的完整表格和不可重入问题

**Q7: ES 和 DB 数据同步方案？增量同步怎么设计？异常怎么补偿？**

📝 **参考答案（完整版）**：

**候选人的实际方案（外呼名单场景）：**

```
原始方案：逐条导入 → 逐条写 ES → 逐条写 DB（性能瓶颈，数据丢失）
    ↓ 优化
优化方案：
1. 解析数据 → 投入消息队列（削峰）
2. 消费端批量聚合（200 条/批 或 1 分钟窗口）— micro-batch 策略

4. 失败数据 → 重试队列（指数退避重试）
```

- 消息队列削峰：解决高 QPS 下的写入瓶颈
- 批量聚合（200 条/批 或 1 分钟）：数量+时间双触发的 micro-batch 策略
- 指数退避重试：避免雪崩式重试
- 兜底链路完整：重试队列 → 落库标记 → 人工一键重导

**优化建议：**
- 写入顺序建议改为：先写 DB（数据源）再写 ES（查询层），保证数据真相在 DB
- 可补充对账任务定期校验 DB 和 ES 数据一致性

👤 **候选人回答**：详细描述了从逐条写入到批量聚合+MQ 削峰的优化过程，包含指数退避重试和人工兜底机制

### 1.4 项目经验深入

**Q8: 短信优化方案——为什么能替代短信？"用户行为动态决策"怎么做的？怎么证明下单量反升 1.8% 的因果关系？**

📝 **参考答案（完整版）**：

**方案核心：微信服务号消息卡片替代短信**

- 不依赖用户订阅消息模板，与下单/接单/取消等订单状态变更绑定
- 用户在微信内时推送服务号消息卡片，消息必达且可见性远高于短信

**为什么能替代短信：**
1. 短信到达率低：用户屏蔽/拦截垃圾短信，实际触达率不可控
2. 微信消息卡片必达：只要用户打开微信就能看到
3. 点击回流：消息卡片可直接跳转小程序，缩短回流路径

**因果关系证明 — AB 实验：**
- 实验组：微信消息卡片推送
- 对照组：传统短信触达
- 跑 2-4 周数据，对比下单转化率、回流率、客诉率
- 实验组下单量提升 1.8%，客诉无波动 → 验证因果关系 → 全量上线

**用户行为动态决策（分层触达策略）：**
- 用户**在小程序内** → 不推送
- 用户**在微信内但不在小程序** → 服务号消息卡片
- 用户**不在微信内** → 兜底发短信

👤 **候选人回答**：清晰讲述了微信消息卡片替代短信的逻辑——绑定订单状态推送、消息必达、短信被屏蔽的场景。以"司机取消订单→用户无感知→流失"vs"消息卡片→感知→回流→重新下单"的因果链论证下单量提升。

📊 **评价**：✅ 业务理解非常到位，因果推理清晰。但证明因果关系更严谨的方式应是 AB 实验，建议主动提及。

💡 **面试官补充讲解**：建议面试时主动补充 AB 实验验证因果关系，体现数据驱动决策思维

**Q9: MCP 化中的 Redis 缓存透传机制？为什么能把下单简化为 1 个 ID？ID 生命周期怎么管理？**

📝 **参考答案（完整版）**：

**方案本质：用 estimateId 做"上下文指针"**

```
询价阶段：BFF 并行调 N 家服务商 → 聚合结果 → 生成 estimateId 
         → 结果写入 Redis（key=estimateId, TTL=5min）
         → 只把 estimateId 返回给 Agent 上下文

下单阶段：Agent 只传 estimateId → BFF 从 Redis 取出完整询价数据 → 发起下单
```

**三大好处：**
1. 节省 Token — 上下文只传一个 ID，不传几十个字段
2. 降低模型理解复杂度 — 模型不需理解每个询价参数含义
3. 数据一致性 — 询价和下单用同一份缓存数据

**TTL 设计**：取所有服务商中最短的缓存失效时间（木桶效应），当前为 5 分钟

**过期处理**：下单时查 Redis，若 estimateId 已过期，返回"报价已失效"，Agent 自动重新询价

👤 **候选人回答**：讲述清晰——estimateId 做上下文指针、节省 Token、TTL 取服务商最短失效时间（5 分钟）、每次询价生成新 ID 写入 Redis、下单时按 ID 取缓存

📊 **评价**：✅ 回答质量很高，方案设计合理，能清晰表达缓存透传的设计思路和节省 Token 的目的

💡 **面试官补充讲解**：补充了过期场景的闭环处理（Agent 自动重新询价），建议面试时主动提及

**Q12: Multi-Agent 架构怎么理解？如何把顺风车从单 Agent 升级为多 Agent？**

📝 **参考答案（完整版）**：

**多 Agent 架构核心概念：**

```
┌─────────────────────────────────────────────┐
│           Router Agent（路由/编排）            │
│   意图识别 → 任务拆分 → 分发子 Agent → 汇总结果 │
└──────────┬──────────┬──────────┬─────────────┘
           │          │          │
    ┌──────▼──┐ ┌─────▼────┐ ┌──▼──────────┐
    │ 地址Agent│ │ 询价Agent │ │ 下单Agent    │
    └─────────┘ └──────────┘ └─────────────┘
```

**关键架构要素：**
1. **Router Agent**：总调度，负责意图识别、任务拆分、子 Agent 分发和结果汇总
2. **子 Agent**：各司其职（地址解析、多品类询价、下单流转、客服处理等）
3. **通信机制**：共享上下文（Blackboard 模式）或消息传递
4. **冲突处理**：核心品类必须成功，非核心超时降级
5. **单 vs 多的取舍**：工具少于 5 个、流程简单时单 Agent 足够；工具多、流程复杂、需要并行时用多 Agent

👤 **候选人回答**：从业务出发提出了并行化提效（地址解析/POI 消歧并行）和多品类聚合推荐（顺风车/打车/城际拼车比价），有"智能出行助手"的产品思维。

📊 **评价**：⚠️ 业务直觉好，并行化和多品类推荐的思路有价值。但缺少 Router Agent 编排层概念、Agent 通信机制（Blackboard 模式）和冲突处理策略。

### 1.7 算法与编程

**Q13: LeetCode 236. 二叉树的最近公共祖先（Medium）**

📝 **参考答案（完整版）**：

**题目**：给定一个二叉树，找到两个指定节点 p、q 的最近公共祖先（LCA）。

**核心思路：后序遍历 DFS**

递归函数含义：在以 root 为根的子树中查找 p 和 q。

返回值三种情况：
- `null`：子树里既没有 p 也没有 q
- `p` 或 `q`：找到了其中一个
- LCA 节点：找到了最近公共祖先

**关键判断逻辑（4 种情况）：**
1. `root == null` → 返回 null（递归终止）
2. `root == p` 或 `root == q` → 返回 root（找到目标）
3. 递归左右子树得 left、right：
   - `left != null && right != null` → p 和 q 分居两侧，root 就是 LCA
   - 只有一侧不为 null → 返回那一侧（LCA 在那侧子树中）

**完整代码：**

```java
class Solution {
    public TreeNode lowestCommonAncestor(TreeNode root, TreeNode p, TreeNode q) {
        // 递归终止：到底了，或者找到了 p/q
        if (root == null || root == p || root == q) {
            return root;
        }
        
        // 后序遍历：先递归左右子树
        TreeNode left = lowestCommonAncestor(root.left, p, q);
        TreeNode right = lowestCommonAncestor(root.right, p, q);
        
        // 左右都不为空 → p 和 q 分居两侧，当前 root 就是 LCA
        if (left != null && right != null) {
            return root;
        }
        // 只有一侧不为空 → LCA 在那一侧
        return left != null ? left : right;
    }
}
```

**复杂度分析：**
- 时间复杂度：O(N)，每个节点访问一次
- 空间复杂度：O(N)，最坏情况树退化为链表，递归栈深度为 N

**扩展：N 叉树的 LCA**

将 `left/right` 两路改为 `for` 循环遍历所有子节点：

```java
List<Node> found = new ArrayList<>();
for (Node child : root.children) {
    Node result = lca(child, p, q);
    if (result != null) found.add(result);
}
if (found.size() >= 2) return root;
return found.isEmpty() ? null : found.get(0);
```

👤 **候选人回答**：
- 思路方向：知道用 DFS 递归，提到了结合二叉树左右子树特性
- 具体逻辑：描述比较模糊，没有清晰说出"后序遍历 + 左右子树返回值判断"的核心逻辑
- 代码编写：未自己写代码，要求面试官给出参考答案
- 复杂度分析：知道时间复杂度 O(N)，但漏了空间复杂度 O(N)
- N 叉树扩展：没有思路

📊 **评价**：⚠️ 方向正确（DFS 递归），但思路表述不够清晰，未能独立写出代码，复杂度分析不完整。算法能力需要加强练习。

💡 **面试官补充讲解**：给出了完整代码（仅 10 行核心逻辑）、4 种判断情况的清晰梳理、复杂度分析、以及 N 叉树扩展方案。建议背下这道经典题。
