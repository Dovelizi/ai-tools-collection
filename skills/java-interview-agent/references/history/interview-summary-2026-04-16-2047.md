### 1.1 Java 基础与 JVM

**Q1: synchronized 和 ReentrantLock 有什么区别？在你的项目中，什么场景下会选择用 ReentrantLock 而不是 synchronized？**

📝 **参考答案（完整版）**：

**synchronized 和 ReentrantLock 的核心区别：**

| 维度 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 实现层面 | JVM 层面（monitorenter/monitorexit 字节码） | Java API 层面（java.util.concurrent.locks） |
| 锁释放 | 自动释放（出作用域或异常） | **手动释放**（必须在 finally 中 unlock） |
| 可中断 | 不可中断（线程会一直等待） | **可中断**（lockInterruptibly()） |
| 超时尝试 | 不支持 | **支持**（tryLock(timeout)） |
| 公平性 | 非公平锁 | 可选**公平锁**（new ReentrantLock(true)） |
| 条件变量 | 只有一个 wait/notify | **多个 Condition**（更灵活） |
| 锁升级 | 无锁→偏向锁→轻量级锁→重量级锁（JDK 6+） | 无锁升级优化 |
| 可重入 | 支持 | 支持 |

**synchronized 锁升级顺序（JDK 6+）：**

```
无锁 → 偏向锁（单线程访问） → 轻量级锁（CAS，少量竞争） → 重量级锁（OS 互斥量，激烈竞争）
```

- 偏向锁：只有一个线程访问，直接偏向该线程，后续无需同步
- 轻量级锁：少量线程竞争，CAS 自旋获取，避免阻塞
- 重量级锁：竞争激烈，线程阻塞挂起，由 OS 调度

**选择原则：**
- **简单同步用 synchronized**：代码简洁、JVM 自动优化、自动释放锁
- **需要高级特性用 ReentrantLock**：超时/中断/公平/多条件等场景

👤 **候选人回答**：
- 提到了 synchronized 锁升级机制 ✅
- 但锁升级顺序说错了——说"先轻量级锁再偏向锁"（应该是先偏向锁再轻量级锁）
- ReentrantLock 只提到"可重入"，没有说出可中断、超时、公平锁、多条件等核心区别
- 使用场景说反了——说"低并发用 ReentrantLock，高并发用 synchronized"（实际应该反过来，复杂场景用 ReentrantLock）

📊 **评价**：⚠️ 知道锁升级的存在但顺序记错（这是第二次了），ReentrantLock 的高级特性没有展开，使用场景判断不准确。

💡 **面试官补充讲解**：纠正了锁升级顺序（无锁→偏向锁→轻量级锁→重量级锁），补充了 ReentrantLock 四大高级特性（可中断/超时/公平/多条件），明确了选择原则。

### 1.2 中间件与数据存储

**Q2（间隔重复-MVCC 换角度）: 在你的项目中，比如卡单一致性校验的场景，MySQL 用的是什么隔离级别？RR 隔离级别下是怎么通过 MVCC 解决"不可重复读"问题的？能结合业务场景说说吗？**

📝 **参考答案（完整版）**：

**RR 解决不可重复读的核心机制：**

ReadView **只在事务第一次 SELECT 时生成一次，后续复用同一个 ReadView**。整个事务期间看到的数据快照是一致的，不管其他事务怎么提交都看不到。

**RC vs RR 的核心区别：**

| 隔离级别 | ReadView 生成时机 | 效果 |
|---------|-----------------|------|
| RC（读已提交） | **每次 SELECT** 都生成新的 ReadView | 能看到其他事务已提交的最新数据 |
| RR（可重复读） | **事务第一次 SELECT** 时生成，后续复用 | 整个事务期间看到的数据一致（快照读） |

一句话记忆：**RC 每次读都拍新照片，RR 只拍一次照片用到底。**

**ReadView 可见性判断规则：**
- `trx_id == creator_trx_id` → 自己修改的，可见
- `trx_id < min_trx_id` → 事务已提交，可见
- `trx_id >= max_trx_id` → 事务在 ReadView 创建后才开始，不可见
- `min_trx_id <= trx_id < max_trx_id` → 看是否在 m_ids 中：在则不可见（未提交），不在则可见（已提交）

**结合卡单业务场景：**
并发场景下读取订单状态时，RR 隔离级别保证了同一个事务内多次读到的状态一致，不会出现中途被其他事务修改导致的逻辑混乱。

👤 **候选人回答**：
- 知道默认是 RR 级别 ✅
- 提到了事务 ID、版本链、活跃事务数组、ReadView 等关键词 ✅（比上次有进步）
- 但可见性规则描述不准确——说"读取最活跃的事务 ID 就是最新数据"（实际是沿版本链找第一个可见版本，活跃事务中的修改恰恰是不可见的）
- 业务场景没有关联上

📊 **评价**：⚠️ 比上次（❌ 完全不了解）有明显进步，能说出关键术语。但可见性规则的核心逻辑还不够准确，RR "只生成一次 ReadView" 这个最关键的点没有明确说出。

💡 **面试官补充讲解**：强调了 RR 只生成一次 ReadView 是核心机制，纠正了可见性规则，补充了卡单业务场景的关联。

### 1.3 Spring 框架

**Q3（间隔重复-@Transactional 换角度）: 假设你在项目中遇到一个 bug：一个 Service 方法加了 @Transactional，方法里抛了 IOException，但事务没有回滚。你怎么排查？能说出至少 4 种 @Transactional 可能失效的场景吗？**

📝 **参考答案（完整版）**：

**@Transactional 失效的六大常见场景：**

1. **同类自调用**：`this.method()` 绕过代理对象，事务失效。解决：注入自身或 `AopContext.currentProxy()`
2. **方法不是 public**：Spring AOP 只能代理 public 方法
3. **非 Spring 管理的对象**：手动 new 的对象没有代理
4. **异常类型不对**：默认只回滚 RuntimeException 和 Error，检查异常（IOException 等）不回滚。需要 `@Transactional(rollbackFor = Exception.class)`
5. **catch 吞掉了异常**：异常被 catch 处理后 Spring 认为没异常，不回滚
6. **数据库引擎不支持事务**：MyISAM 不支持，只有 InnoDB 支持

**本题答案：** 抛了 IOException 但没回滚，原因是 IOException 是检查异常，不在 @Transactional 默认的回滚范围（RuntimeException + Error）内。修复：`@Transactional(rollbackFor = Exception.class)`。

👤 **候选人回答**：
- 自调用 ✅
- 非 public 方法 ✅
- 非 Spring 管理的对象 ✅
- 异常被 try-catch 吞掉 ✅
- 异常类型不对 ⚠️（方向对但细节说反了——说"标注 Exception 但抛 IOException 不捕捉"，实际 IOException 是 Exception 子类，如果标了 rollbackFor=Exception.class 反而能捕获。问题出在没标 rollbackFor 用了默认配置）
- 排查思路：先看 @Transactional 注解里的异常配置 ✅

📊 **评价**：⚠️ 比上次（只答 2 个）进步很大，一口气说了 5 个，其中 4 个准确。异常类型的细节需要更精确——默认只回滚 RuntimeException 和 Error，检查异常不回滚。

💡 **面试官补充讲解**：纠正了异常类型的精确描述，补充了数据库引擎不支持事务的场景。

### 1.7 算法与编程

**Q9: LeetCode 200. 岛屿数量（Medium）**

📝 **参考答案（完整版）**：

**题目**：给定 '1'（陆地）和 '0'（水）组成的二维网格，计算岛屿数量。

**核心思路：DFS（深度优先搜索）— "沉岛法"**

1. 遍历整个二维网格
2. 每发现一个 '1'，岛屿计数 +1
3. 从这个 '1' 出发，DFS 把所有相连的 '1' 全部标记为 '0'（沉掉这座岛）
4. 下次遍历不会重复计算同一座岛

**完整代码：**

```java
class Solution {
    public int numIslands(char[][] grid) {
        int count = 0;
        for (int i = 0; i < grid.length; i++) {
            for (int j = 0; j < grid[0].length; j++) {
                if (grid[i][j] == '1') {
                    count++;          // 发现新岛屿
                    dfs(grid, i, j);  // 沉掉整座岛
                }
            }
        }
        return count;
    }

    private void dfs(char[][] grid, int i, int j) {
        if (i < 0 || i >= grid.length || j < 0 || j >= grid[0].length 
            || grid[i][j] == '0') {
            return;
        }
        grid[i][j] = '0'; // 标记为已访问
        dfs(grid, i + 1, j); // 下
        dfs(grid, i - 1, j); // 上
        dfs(grid, i, j + 1); // 右
        dfs(grid, i, j - 1); // 左
    }
}
```

**复杂度分析：**
- 时间复杂度：O(M × N)，每个格子最多访问一次
- 空间复杂度：O(M × N)，最坏情况递归栈深度（全是陆地）

**题型分类提醒：**
- 滑动窗口：一维数组/字符串连续子序列问题
- DFS/BFS：二维网格搜索、树遍历、图遍历
- 双指针：有序数组、链表问题

👤 **候选人回答**：
- 提出用滑动窗口解决 ❌（滑动窗口用于一维连续子序列，不适用于二维不规则网格搜索）
- 描述"二维滑动窗口横向移再纵向移"——岛屿形状不规则，矩形窗口无法框住 L 形等异形岛屿

📊 **评价**：❌ 题型识别错误。岛屿数量是经典的 DFS/BFS 网格搜索题，不是滑动窗口。需要按题型分类练习，区分不同算法的适用场景。

💡 **面试官补充讲解**：给出了 DFS "沉岛法"完整代码，讲解了滑动窗口 vs DFS/BFS vs 双指针的适用场景区分。
