### 1.1 Java 基础与 JVM

**Q1: ThreadLocal 的底层数据结构是什么？在线程池场景下使用 ThreadLocal 有什么风险？怎么解决？**

📝 **参考答案（完整版）**：

**底层数据结构：**

ThreadLocal 的数据存储在每个线程的 `ThreadLocalMap` 中。ThreadLocalMap 是一个定制化的哈希表，使用**开放地址法**解决冲突（不是链表法）。

```
Thread 对象
  └── ThreadLocal.ThreadLocalMap threadLocals
        └── Entry[] table
              └── Entry(弱引用的 ThreadLocal key, Object value)
```

- Key：ThreadLocal 实例本身（**弱引用** WeakReference）
- Value：用户 set 进去的值（**强引用**）

**线程池场景下的内存泄漏原因：**

1. 线程池中的线程是**复用的**，线程不会销毁 → ThreadLocalMap 也不会被回收
2. 当 ThreadLocal 对象被 GC 回收后（因为 Key 是弱引用），Key 变成 null
3. 但 **Value 仍然被 Entry 强引用着**，无法被 GC 回收
4. 形成 Key=null 但 Value 无法回收的**"脏数据"**，随着线程复用不断累积 → 内存泄漏

**另一个风险——脏数据：**
- 线程池线程复用时，上一个任务 set 的 ThreadLocal 值可能被下一个任务读到
- 导致数据错乱（如用户 A 的请求读到用户 B 的数据）

**解决方案：**
```java
try {
    threadLocal.set(value);
    // 业务逻辑
} finally {
    threadLocal.remove(); // 必须在 finally 中清理
}
```

核心原则：**用完必须 remove()，尤其在线程池场景下是强制要求**。

👤 **候选人回答**：知道底层是哈希表结构，提到线程池下有内存泄漏风险，但将原因说成"多线程访问"（实际 ThreadLocal 是线程隔离的），具体解决方案不清楚。

📊 **评价**：⚠️ 方向正确但关键原因不准确。ThreadLocal 本身就是线程隔离的，泄漏原因是 Key 弱引用被 GC 但 Value 强引用未释放 + 线程池线程不销毁。

💡 **面试官补充讲解**：纠正了泄漏原因，讲解了 Key 弱引用 + Value 强引用的不对称性，强调 finally 中 remove() 是必须的。

**Q2: JDK 21 引入的虚拟线程（Virtual Threads）了解吗？和传统平台线程的区别？适合什么场景？**

📝 **参考答案（完整版）**：

**虚拟线程**是 JDK 21 正式发布的特性（Project Loom），本质是 **JVM 层面的协程**。

| 维度 | 平台线程（传统） | 虚拟线程（JDK 21） |
|------|------|------|
| 调度方 | **OS 内核**调度 | **JVM** 调度 |
| 内存开销 | 每个线程约 **1MB** 栈空间 | 约 **几KB**，非常轻量 |
| 数量上限 | 几千个就吃力 | 可以轻松创建 **百万级** |
| 阻塞代价 | 阻塞 = 占用 OS 线程 | 阻塞时自动**挂起**（unmount），释放底层平台线程（carrier thread） |
| 创建方式 | `new Thread()` / 线程池 | `Thread.ofVirtual().start()` / `Executors.newVirtualThreadPerTaskExecutor()` |

**核心优势**：用同步代码写出异步性能。虚拟线程遇到 IO 阻塞（网络/磁盘/DB）时自动挂起，底层平台线程可以去执行其他虚拟线程。

**适合场景**：IO 密集型——大量 HTTP 调用、数据库查询、外呼系统等。

**不适合场景**：CPU 密集型计算、synchronized 块内的阻塞（会 pin 住 carrier thread）。

**使用注意**：
- 不需要池化虚拟线程（一个任务一个虚拟线程）
- 避免在虚拟线程中使用 synchronized，改用 ReentrantLock
- ThreadLocal 在虚拟线程中慎用（百万级线程 × ThreadLocal = 内存爆炸），推荐 ScopedValues

👤 **候选人回答**：不了解

📊 **评价**：❌ 未掌握。JDK 21 虚拟线程是 2025-2026 面试热点，建议花 1 小时了解核心概念。

💡 **面试官补充讲解**：讲解了虚拟线程 vs 平台线程对比表、适用场景和使用注意事项。

### 1.2 Spring 框架

**Q3: Spring AOP 的底层实现原理？JDK 动态代理和 CGLIB 分别在什么时候使用？项目中有用到 AOP 的实际场景吗？**

📝 **参考答案（完整版）**：

**AOP 底层实现**：Spring AOP 基于动态代理，在 Bean 初始化阶段的 `BeanPostProcessor.postProcessAfterInitialization()` 中，由 `AbstractAutoProxyCreator` 判断是否需要创建代理对象。

**JDK 动态代理 vs CGLIB：**

| 维度 | JDK 动态代理 | CGLIB |
|------|------|------|
| 前提条件 | 目标类**必须实现接口** | **不需要**实现接口 |
| 实现方式 | 基于 `java.lang.reflect.Proxy` + `InvocationHandler` | 基于**字节码生成子类**（ASM 库） |
| 限制 | 只能代理接口方法 | 不能代理 **final 类和 final 方法** |
| 性能 | JDK 8+ 性能已优化，差距不大 | 生成子类有额外开销 |

**Spring Boot 2.x+ 默认行为**：`spring.aop.proxy-target-class=true`，默认使用 **CGLIB**，不管有没有实现接口。

**AOP 典型应用场景（横切关注点）：**
1. **事务管理**：@Transactional
2. **日志记录**：方法调用前后打印入参出参
3. **权限校验**：自定义 @RequireLogin 注解
4. **接口耗时统计**：@Around 记录方法执行时间
5. **缓存切面**：@Cacheable

**五种通知类型**：@Before、@After、@AfterReturning、@AfterThrowing、@Around

👤 **候选人回答**：
- 知道 AOP 底层基于 JDK 动态代理和 CGLIB ✅
- JDK 动态代理只能代理实现接口的类 ✅
- CGLIB 用于没实现接口的类 ✅（但说了"代理私有方法"不准确）
- 举例 @Transactional 作为 AOP 场景 ✅
- 提到 Bean 属性修改和 MyBatis Plus 注入也是 AOP（不够准确，偏向 BeanPostProcessor）

📊 **评价**：⚠️ 核心区别答对了，@Transactional 举例恰当。但 CGLIB 限制说成"代理私有方法"不准确（应是不能代理 final），Bean 属性修改不属于 AOP 范畴。

💡 **面试官补充讲解**：纠正了 CGLIB 限制（final 而非 private），补充了 Spring Boot 2.x+ 默认用 CGLIB 的行为，区分了 AOP 和 BeanPostProcessor 的职责。

### 1.3 中间件与数据存储

**Q4: RocketMQ 中消息堆积了怎么处理？线上已经堆积了几百万条消息，怎么紧急处理？**

📝 **参考答案（完整版）**：

**紧急处理三步走：**

**第一步：快速定位原因（止血）**
- 消费者是报错了（异常导致重试死循环）？还是消费太慢（下游 DB/接口慢）？
- 查看消费者日志 + 监控（消费 RT、失败率、下游依赖健康度）
- 如果是 Bug → **先修 Bug 再扩容**，否则扩容也没用

**第二步：紧急扩容消费能力**
- **关键点**：RocketMQ 消费并行度 = min(消费者实例数, Queue 数)
- 如果 Queue 数不够 → 临时新建 Topic（更多 Queue），写转发消费者把堆积消息快速转发
- 部署 N 倍消费者实例并行消费
- 消费端开启**批量消费**（`consumeMessageBatchMaxSize`），一次拉多条处理

**第三步：根因修复 + 预防**
- 优化消费逻辑（批量消费、异步化、减少下游调用）
- 配置消费堆积告警（堆积超阈值自动报警）
- 评估消费能力是否匹配生产能力，做好容量规划
- 考虑消费降级策略：非核心消息可临时丢弃或延后处理

👤 **候选人回答**：提到先定位原因，再横向扩容消费机器。区分了需要实时性（扩容）和不需要实时性（削峰）两种场景。

📊 **评价**：⚠️ 思路方向正确，但缺少关键点：Queue 数必须匹配消费者实例数（否则扩机器无效）、批量消费优化、转发到新 Topic 的应急方案。

💡 **面试官补充讲解**：补充了 Queue 数与消费并行度的关系、转发新 Topic 方案和容量规划。

**Q5: MySQL 的 MVCC（多版本并发控制）是怎么实现的？ReadView 的生成时机在 RC 和 RR 隔离级别下有什么区别？**

📝 **参考答案（完整版）**：

**MVCC 实现的三大核心组件：**

1. **隐藏字段**：每行数据有两个隐藏列
   - `DB_TRX_ID`：最近修改该行的事务 ID
   - `DB_ROLL_PTR`：回滚指针，指向 Undo Log 中该行的上一个版本

2. **Undo Log（版本链）**：每次 UPDATE 都在 Undo Log 中保留旧版本，通过 `DB_ROLL_PTR` 串成一条**版本链**

3. **ReadView（快照）**：事务在读数据时生成的"可见性判断规则"，包含：
   - `m_ids`：当前活跃（未提交）的事务 ID 列表
   - `min_trx_id`：活跃事务中最小的 ID
   - `max_trx_id`：下一个将要分配的事务 ID
   - `creator_trx_id`：创建该 ReadView 的事务 ID

**读取规则**：沿着版本链从最新版本往前找，找到第一个"对当前 ReadView 可见"的版本返回。

**可见性判断**：
- `trx_id == creator_trx_id` → 自己修改的，可见
- `trx_id < min_trx_id` → 事务已提交，可见
- `trx_id >= max_trx_id` → 事务在 ReadView 创建后才开始，不可见
- `min_trx_id <= trx_id < max_trx_id` → 看是否在 m_ids 中：在则不可见（未提交），不在则可见（已提交）

**RC vs RR 核心区别：**

| 隔离级别 | ReadView 生成时机 | 效果 |
|---------|---------|---------|
| **RC（读已提交）** | **每次 SELECT** 都生成新的 ReadView | 能看到其他事务已提交的最新数据 |
| **RR（可重复读）** | **事务第一次 SELECT** 时生成，后续复用 | 整个事务期间看到的数据一致（快照读） |

一句话记忆：**RC 每次读都拍新照片，RR 只拍一次照片用到底**。

👤 **候选人回答**：不了解

📊 **评价**：❌ 未掌握。MVCC 是 MySQL 大厂必考高频题，ReadView + Undo Log 版本链必须掌握。

💡 **面试官补充讲解**：完整讲解了 MVCC 三大组件、版本链读取规则和 RC/RR 的 ReadView 生成时机区别。

### 1.4 项目经验深入

**Q6: 卡单方案——根因是什么？一致性方案怎么设计的？怎么做到客诉趋近于 0？**

📝 **参考答案（面试官视角）**：

**根因**：服务商状态回调缺失（网络异常/服务商故障）→ 状态跳变被拦截（如不能从"派单中"直接跳到"已完单"）→ 订单状态卡住，用户看到的状态与实际不一致。

**一致性方案——主动+被动双通道：**

```
主动通道：用户进入订单详情页 → 前 N 次主动拉取服务商最新状态 → 同步更新
           解决：历史存量卡单 + 实时补偿

被动通道：接单后发延迟消息 → 消费时拉取服务商状态 → 同步更新
           采用分段延迟策略：
           - 接单→行程中：短周期（固定间隔）
           - 行程中→完单：按里程动态计算（里程÷基准距离×时间刻度）
```

**为什么趋近于 0 而非等于 0**：历史归档订单（如 2021 年）服务商接口查不到数据，无法更新。

**设计哲学**：被动消息保证覆盖率，主动拉取保证实时性，两者互为兜底。

👤 **候选人回答**：非常完整清晰地讲述了根因（服务商回调缺失→状态跳变拦截→卡住）、双通道方案（主动拉取+被动延迟消息）、分段延迟策略（短途高频/长途按里程计算）、趋近于 0 的边界 case（历史归档数据）

📊 **评价**：✅ 回答质量很高。根因分析准确，方案设计完整，分段延迟策略有深度，主动承认边界 case 体现了工程严谨性。

💡 **面试官补充讲解**：建议面试时补充数据量化（上线前后客诉率对比），增强说服力。

### 1.7 算法与编程

**Q10: LeetCode 3. 无重复字符的最长子串（Medium）**

📝 **参考答案（完整版）**：

**题目**：给定字符串 s，找出不含重复字符的最长子串的长度。

**核心思路：滑动窗口（双指针 + HashSet/HashMap）**

左指针 left、右指针 right，维护一个无重复字符的窗口：
- right 不断右移扩大窗口
- 发现重复字符时，left 右移缩小窗口，直到窗口内无重复
- 每次更新 max(窗口长度)

**方案一：HashSet 判重**

```java
class Solution {
    public int lengthOfLongestSubstring(String s) {
        Set<Character> set = new HashSet<>();
        int left = 0, maxLen = 0;
        for (int right = 0; right < s.length(); right++) {
            while (set.contains(s.charAt(right))) {
                set.remove(s.charAt(left));
                left++;
            }
            set.add(s.charAt(right));
            maxLen = Math.max(maxLen, right - left + 1);
        }
        return maxLen;
    }
}
```

**方案二：HashMap 记录位置（更优，left 可以直接跳）**

```java
class Solution {
    public int lengthOfLongestSubstring(String s) {
        Map<Character, Integer> map = new HashMap<>();
        int left = 0, maxLen = 0;
        for (int right = 0; right < s.length(); right++) {
            if (map.containsKey(s.charAt(right))) {
                left = Math.max(left, map.get(s.charAt(right)) + 1);
            }
            map.put(s.charAt(right), right);
            maxLen = Math.max(maxLen, right - left + 1);
        }
        return maxLen;
    }
}
```

**复杂度分析：**
- 时间复杂度：O(n)，每个字符最多被访问两次
- 空间复杂度：O(min(n, 字符集大小))

**关键点**：滑动窗口题标准模板是**双指针同向移动、窗口大小动态变化**，不是固定窗口从大到小缩减。
