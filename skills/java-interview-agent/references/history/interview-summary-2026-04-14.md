### 1.1 Java 基础与 JVM（历史薄弱项抽查）

**Q1: HashMap 在 JDK 7 和 JDK 8 中有哪些关键区别？JDK 7 头插法在多线程扩容时导致什么问题？**

📝 **参考答案（完整版）**：

| 对比项 | JDK 7 | JDK 8 |
|--------|-------|-------|
| 数据结构 | 数组 + 链表 | 数组 + 链表 + **红黑树** |
| 插入方式 | **头插法** | **尾插法** |
| hash 计算 | 4次位运算 + 5次异或 | 1次位运算 + 1次异或（更简洁） |
| 扩容迁移 | 全部 rehash，链表反转 | 高低位链拆分（`hash & oldCap`），保持顺序 |

**树化条件（双条件）**：链表长度 ≥ **8** 且数组长度 ≥ **64** 时转红黑树；链表长度 ≥ 8 但数组长度 < 64 时优先扩容。红黑树节点 ≤ 6 时退化为链表。

**头插法的致命问题 — 并发扩容导致环形链表：**
- 假设链表 A → B → C，两个线程同时扩容
- 头插法会反转链表顺序
- 线程 A 执行到一半挂起，线程 B 完成扩容（顺序变为 C → B → A）
- 线程 A 恢复执行，按旧的指针关系操作，导致 A.next = B，B.next = A
- 形成**环形链表**，后续 `get()` 遍历到此处陷入**死循环**，CPU 100%

JDK 8 改为尾插法 + 高低位链拆分，扩容时保持原有顺序，避免此问题。

👤 **候选人回答**：知道 JDK 7 数组+链表、JDK 8 数组+链表+红黑树；链表≥8 转红黑树、≤6 退化链表；头插法并发扩容导致环形链表、CPU 100%

📊 **评价**：⚠️ 核心要点都答到了，比前两次有显著进步。遗漏点：树化双条件（还需要数组≥64）、JDK 8 改为尾插法、高低位链拆分

💡 **面试官补充讲解**：补充了树化双条件、尾插法、高低位链拆分迁移方式

**Q2: ConcurrentHashMap 在 JDK 7 和 JDK 8 中的实现区别？为什么放弃分段锁？**

📝 **参考答案（完整版）**：

**JDK 7 — 分段锁（Segment）**：
- 内部将数据分成 16 个 Segment（默认），每个 Segment 继承自 ReentrantLock
- 每次操作只锁一个 Segment，不同 Segment 可以并发访问
- 并发度 = Segment 数量（默认 16）

**JDK 8 — CAS + synchronized（Node 级别锁）**：
- 结构变成和 HashMap 一样的数组 + 链表 + 红黑树
- 空桶时用 CAS 直接写入，无需加锁
- 非空桶时对头节点加 synchronized，锁粒度从 Segment 级别细化到单个桶（Node）级别

**为什么放弃分段锁：**
1. 锁粒度更细：Segment 锁一锁一片桶，Node 锁只锁一个桶，并发度大幅提升
2. 内存开销更小：不需要维护 Segment 数组和额外的 ReentrantLock 对象
3. JVM 对 synchronized 做了大量优化（偏向锁→轻量级锁→重量级锁），低竞争下性能不输 ReentrantLock

记忆口诀：**JDK 7 锁 Segment（一片），JDK 8 锁 Node（一个桶），粒度更细并发更高**

👤 **候选人回答**：JDK 7 用 16 个 Segment 并发控制，使用 ReentrantLock；JDK 8 用 CAS + synchronized；放弃分段锁因为资源消耗大 + JDK 8 对 synchronized 做了锁升级优化

📊 **评价**：⚠️ 比上次"完全不了解"有很大进步，核心要点答到。但锁粒度描述不够精准（说"锁整个节点"不准确，应该是 Segment 锁一片桶 vs Node 锁单个桶）

💡 **面试官补充讲解**：纠正了锁粒度的精确表述，补充了放弃分段锁的三个原因

**Q3: G1 垃圾收集器的工作原理？和 CMS 相比有哪些核心优势？**

📝 **参考答案（完整版）**：

**G1 核心思想**：把堆划分为约 2048 个大小相等的 Region，每个 Region 动态扮演 Eden/Survivor/Old/Humongous 角色。

**四个阶段**：
1. 初始标记（STW）：标记 GC Roots 直接引用的对象
2. 并发标记：与用户线程并发，遍历对象图找存活对象
3. 最终标记（STW）：处理并发标记期间的变更（SATB 快照算法）
4. 筛选回收（STW）：按回收价值排序，优先回收垃圾最多的 Region（Garbage-First 名字由来）

**G1 关键特性 — 可预测停顿时间模型**：
- `-XX:MaxGCPauseMillis=200` 设置期望停顿时间
- G1 在期望时间内选择回收收益最大的 Region 集合

**G1 vs CMS 对比**：

| 维度 | CMS | G1 |
|------|-----|-----|
| 内存布局 | 固定分代 | Region 动态分代 |
| 回收算法 | 标记-清除（产生碎片） | 标记-整理/复制（无碎片） |
| 停顿控制 | 不可预测 | 可设置目标停顿时间 |
| 适用场景 | 堆 < 8G | 堆 ≥ 4G |
| JDK 状态 | JDK 14 被移除 | JDK 9+ 默认收集器 |

记忆口诀：**G1 三板斧 — Region 分区、优先回收、可预测停顿**

👤 **候选人回答**：知道 2048 个 Region、动态分布在各区域；提到标记整理再清理；知道可预测停顿和内存消耗优于 CMS

📊 **评价**：⚠️ 比上次"完全不了解"有进步，知道 Region 分区和可预测停顿。但四个阶段描述不够结构化，G1 vs CMS 对比不够精确（最大优势是无碎片+可预测停顿，不只是内存消耗小）

💡 **面试官补充讲解**：补充了四个阶段详细流程、G1 vs CMS 五维度对比表和记忆口诀

### 1.2 Spring 框架

**Q4: @Transactional 注解在什么情况下会失效？实际项目中有没有踩过坑？**

📝 **参考答案（完整版）**：

@Transactional 失效的六大常见场景：

1. **同类自调用**：`this.method()` 绕过代理对象，事务失效。解决：注入自身或 `AopContext.currentProxy()`
2. **方法不是 public**：Spring AOP 只能代理 public 方法
3. **异常类型不对**：默认只回滚 RuntimeException 和 Error，检查异常（IOException 等）不回滚。需要 `@Transactional(rollbackFor = Exception.class)`
4. **非 Spring 管理的对象**：手动 new 的对象没有代理
5. **catch 吞掉了异常**：异常被 catch 处理后 Spring 认为没异常，不回滚
6. **数据库引擎不支持事务**：MyISAM 不支持，只有 InnoDB 支持

最高频三个：**自调用、异常类型不对、catch 吞异常**

👤 **候选人回答**：提到自调用和非 Bean 对象两种情况，实际项目中没遇到过

📊 **评价**：⚠️ 方向正确但不够全面，只答出 6 个中的 2 个。实战案例缺失。面试时建议至少答出 4 个，最好结合项目举例

### 1.3 中间件与数据存储

**Q5: Redis 分布式锁怎么实现？锁过期了业务没执行完怎么办？Redisson 看门狗机制？**

📝 **参考答案（完整版）**：

**基本实现**：
- 加锁：`SET key value NX EX 30`（NX 互斥，EX 防死锁，value 用 UUID+线程ID 防误删）
- 解锁：Lua 脚本保证原子性（先判断 value 是否自己的，再删除）

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

**Redisson 看门狗机制**：
- 加锁成功后启动后台线程，每隔 10 秒（30s÷3）检查持锁线程是否还在执行
- 还在执行 → 自动续期到 30 秒
- 执行完毕或宕机 → 看门狗停止，锁自然过期释放
- 注意：只有不手动设置 leaseTime 时看门狗才会启动

**加分项**：
- 可重入锁：Hash 结构，field=线程标识，value=重入次数
- RedLock：N 个独立节点，超半数加锁成功才算成功

👤 **候选人回答**：SET key value NX EX 30 加锁；Lua 脚本解锁判断 key 是否存在再删除；看门狗每 10 秒检查、自动续期到 30 秒

📊 **评价**：✅ 核心要点全部答到，比上次"完全不了解"有显著进步。细节可完善：value 应用 UUID 防误删、Lua 解锁是判断 value 而非 key

💡 **面试官补充讲解**：补充了 UUID 防误删、Lua 脚本完整逻辑、看门狗触发条件、可重入和 RedLock

**Q6: MySQL 索引下推（ICP）了解吗？解决什么问题？**

📝 **参考答案（完整版）**：

**索引下推（Index Condition Pushdown, ICP）**：MySQL 5.6+ 的优化，把原本在 Server 层做的条件过滤"下推"到存储引擎层，在索引中提前过滤，减少无效回表次数。

**示例**：联合索引 `(name, age)`，查询 `WHERE name LIKE '李%' AND age = 25`

- 无 ICP：找到所有 name LIKE '李%' 的 1000 条 → 全部回表 → Server 层用 age=25 过滤 → 剩 10 条（浪费 990 次回表）
- 有 ICP：找到 name LIKE '李%' → 在索引层直接用 age=25 过滤 → 只有 10 条回表（减少 990 次无效回表）

**索引下推 vs 覆盖索引：**

| 概念 | 解决方式 | 效果 |
|------|---------|------|
| 覆盖索引 | 查询的列都在索引中 | 完全不回表 |
| 索引下推 ICP | 在索引层提前过滤 | 减少回表次数 |

EXPLAIN 中 `Extra` 列出现 `Using index condition` 就是用了 ICP。

👤 **候选人回答**：将索引下推混淆为覆盖索引（"命中主键索引不需要回表"）

📊 **评价**：❌ 概念混淆。索引下推是"在索引层提前过滤减少回表"，覆盖索引是"查询列都在索引中完全不回表"，两者不同

💡 **面试官补充讲解**：详细讲解了 ICP 原理、有无 ICP 的对比示例和与覆盖索引的区别

### 1.7 算法与编程

**Q11: LeetCode 215. 数组中的第 K 个最大元素 — 要求时间复杂度优于 O(n log n)**

📝 **参考答案（完整版）**：

**方案一：快速选择（QuickSelect）— 最优解**

基于快排的 partition 思想，每次只处理一半，平均时间 O(n)：

```java
public int findKthLargest(int[] nums, int k) {
    int target = nums.length - k;
    int lo = 0, hi = nums.length - 1;
    while (lo < hi) {
        int pivot = partition(nums, lo, hi);
        if (pivot == target) return nums[pivot];
        else if (pivot < target) lo = pivot + 1;
        else hi = pivot - 1;
    }
    return nums[lo];
}

private int partition(int[] nums, int lo, int hi) {
    int pivot = nums[lo];
    int i = lo, j = hi + 1;
    while (true) {
        while (++i <= hi && nums[i] < pivot);
        while (--j >= lo && nums[j] > pivot);
        if (i >= j) break;
        swap(nums, i, j);
    }
    swap(nums, lo, j);
    return j;
}
```
- 时间：平均 O(n)，最坏 O(n²)（随机选 pivot 可优化）
- 空间：O(1)

**方案二：小顶堆 — 简单稳妥**

维护大小为 K 的小顶堆，堆顶就是第 K 大：

```java
public int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> heap = new PriorityQueue<>(k);
    for (int num : nums) {
        heap.offer(num);
        if (heap.size() > k) heap.poll();
    }
    return heap.peek();
}
```
- 时间：O(n log k)
- 空间：O(k)

👤 **候选人回答**：提出桶排序/计数排序思路——用值作为数组下标，出现次数作为值，从后往前遍历找第 K 大。思路方向有一定道理，但未考虑值域过大的问题（如元素值为 10 亿时数组会非常大），且未提及快速选择和小顶堆两种标准解法。

📊 **评价**：⚠️ 有独立思考能力，桶排序思路在值域小的情况下可行，但不是通用解。需要掌握小顶堆（5 行代码，简单稳妥）和快速选择（最优解）两种标准面试解法。

💡 **面试官补充讲解**：提供了快速选择和小顶堆两种完整代码实现，建议面试时先说小顶堆（不容易写错）再提快速选择（展示最优解认知）。
