### 1.1 Java 基础与 JVM

**Q1: JDK 21 虚拟线程（Virtual Threads）的原理？与传统平台线程的区别？优势和坑？**

📝 **参考答案（完整版）**：

**虚拟线程的本质**：
- 虚拟线程是 **JVM 管理的轻量级线程**，由 JVM 调度到 carrier 线程（平台线程）上执行
- 传统平台线程是 **OS 内核线程**，1:1 映射到内核，创建和上下文切换开销大（默认栈 1MB）
- 虚拟线程是 **M:N 模型**（M 个虚拟线程跑在 N 个平台线程上），创建成本极低（类似 Goroutine）

**核心优势**：
- IO 密集型任务场景优势最明显：虚拟线程在 IO 阻塞时会自动让出 carrier 线程（yield），不像传统线程会一直占用 OS 线程
- 可以轻松创建数十万甚至百万级虚拟线程
- 简化并发编程：不再需要异步回调/Reactive，可以用同步代码写高并发

**注意事项（面试必答坑）**：
1. **不要池化虚拟线程**：虚拟线程创建成本极低，应该每次创建新实例（`Thread.startVirtualThread()` 或 `Executors.newVirtualThreadPerTaskExecutor()`），而不是用线程池复用
2. **synchronized 会 pinned**：虚拟线程在 synchronized 块中无法让出 carrier 线程，会 pinned（钉住），导致失去轻量优势。应该改用 `ReentrantLock`
3. **ThreadLocal 慎用**：虚拟线程数量可能极大（百万级），ThreadLocal 会随着每个虚拟线程创建而创建，内存开销巨大

👤 **候选人回答**：知道虚拟线程由用户控制，传统线程由 CPU/OS 调度；虚拟线程性能更好；设置不合理可能引发并发问题

📊 **评价**：⚠️ 方向大致正确，但不够精确。虚拟线程并非"用户控制"，而是 JVM 调度到 carrier 线程。未提到 M:N 模型、pinned 问题、ThreadLocal 慎用等面试核心点

💡 **面试官补充讲解**：补充了虚拟线程本质（M:N 模型）、核心优势场景（IO 密集型）、三大注意点（不要池化、synchronized pinned、ThreadLocal 慎用）

**Q2: ThreadLocal 底层原理？使用不当导致什么问题？**

📝 **参考答案（完整版）**：

**底层原理**：
- 每个 Thread 对象内部有一个 `ThreadLocalMap`（Thread 类的成员变量）
- `ThreadLocalMap` 的 key 是 ThreadLocal 实例（**弱引用**），value 是用户存储的值
- 所以存取逻辑是：当前线程 → 拿到自己的 ThreadLocalMap → 以当前 ThreadLocal 对象为 key 存取 value

**常见误解纠正**：
- ❌ "key 是当前线程" — 不对，key 是 ThreadLocal 实例本身
- ✅ key 是 ThreadLocal 对象（弱引用），每个线程用自己的 ThreadLocalMap 隔离数据

**内存泄漏原因**：
1. ThreadLocal 对象失去外部强引用 → GC 时 key（弱引用）被回收 → key 变成 null
2. 但 value 仍然是强引用链：`Thread → ThreadLocalMap → Entry → value`
3. 如果线程长期存活（如线程池核心线程），value 永远不会被回收
4. 多次操作后，null-key 的 Entry 越来越多，导致内存泄漏

**解决方法**：使用后在 finally 块中调用 `remove()` 清除，或者用 `try-finally-remove` 模式

👤 **候选人回答**：知道 ThreadLocal 底层是特殊 HashMap；key 是弱引用指向当前线程；线程池复用导致 value 无法回收引发内存泄漏

📊 **评价**：⚠️ 核心要点答到，但表述有偏差（"key 是当前线程"应为"key 是 ThreadLocal 实例，当前线程持有一个 ThreadLocalMap"）。内存泄漏原因理解基本正确

💡 **面试官补充讲解**：纠正了"key 是当前线程"的表述，补充了强引用链和 `remove()` 解决方案

**Q3: 类加载机制和双亲委派模型？**

📝 **参考答案（完整版）**：

**类加载 5 步**：加载 → 验证 → 准备 → 解析（可能延迟到初始化后）→ 初始化

**双亲委派模型**：
```
启动类加载器（Bootstrap ClassLoader）— C++ 实现，加载 rt.jar / java.*
    ↑ 委派
扩展类加载器（Extension ClassLoader）— 加载 jre/lib/ext/*
    ↑ 委派
应用程序类加载器（Application ClassLoader）— 加载 classpath 下的类
    ↑ 委派
用户自定义类加载器
```

**工作流程**："自下而上查找，自上而下加载"
1. 类加载器收到请求 → 先委派给父加载器
2. 每一层往上委派，直到 Bootstrap ClassLoader
3. 顶层找不到，子加载器才自己尝试加载

**作用**：保证核心类安全性。如自己写 `java.lang.String`，双亲委派保证由 Bootstrap 加载真正的 String，防止恶意代码替换核心类

**打破双亲委派的场景**：
1. **Tomcat**：Web 应用类加载器优先加载 `WEB-INF/lib/` 下的类，保证多 Web 应用隔离（先自己加载，再委派父加载器）
2. **SPI（JDBC）**：DriverManager 在 Bootstrap 层面，但需要加载第三方驱动，通过 `Thread.currentThread().setContextClassLoader()` 让父加载器获取子加载器去加载

👤 **候选人回答**：了解类加载自上而下扫描；双亲委派打破自下而上顺序可以指定加载；未遇到过打破场景，主动让面试官详细介绍

📊 **评价**：⚠️ 概念方向对，但"自上而下/自下而上"表述比较混乱。双亲委派的核心是"自下而上委派，自上而下加载"。主动询问学习态度值得肯定

💡 **面试官补充讲解**：系统讲解了完整的类加载流程、双亲委派模型层级、两大作用（安全性/核心类保护）、两个打破场景（Tomcat/SPI）

### 1.2 Spring 框架

**Q4: @Async 底层原理？同类自调用为什么失效？**

📝 **参考答案（完整版）**：

**底层原理**：Spring AOP 代理实现
1. Spring 通过 AOP 创建代理对象，代理中有线程池执行逻辑
2. 调用 @Async 方法时，代理对象将方法提交到线程池异步执行

**同类自调用失效原因**：
- `this.method()` 调用的是原始对象的方法，不是代理对象的方法
- AOP 代理只有在外部调用时才会触发
- 解决方法：注入自身（@Autowired）或 `AopContext.currentProxy()`

**生产环境关键点 — 线程池配置**：
- @Async 默认使用 `SimpleAsyncTaskExecutor`，它**不重用线程**，每次创建新线程
- 高并发下可能导致大量线程创建，必须自定义线程池
- 推荐配置：`ThreadPoolTaskExecutor` + 自定义线程名前缀 + CallerRunsPolicy

```java
@Bean("customExecutor")
public Executor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setMaxPoolSize(50);
    executor.setQueueCapacity(200);
    executor.setThreadNamePrefix("async-");
    executor.initialize();
    return executor;
}
```

👤 **候选人回答**：底层基于 AOP 实现；自调用绕过代理导致失效

📊 **评价**：✅ 核心要点正确。可补充：默认线程池不重用线程的问题、自定义线程池的推荐配置

💡 **面试官补充讲解**：补充了 `SimpleAsyncTaskExecutor` 不重用线程的问题和 `ThreadPoolTaskExecutor` 自定义线程池配置

**Q5: Spring 事务传播机制？项目中的使用场景？**

📝 **参考答案（完整版）**：

**七种传播机制**：

| 传播行为 | 说明 |
|---------|------|
| REQUIRED（默认） | 有事务则加入，无则新建 |
| REQUIRES_NEW | 必须新建事务，外层挂起 |
| SUPPORTS | 有则加入，无则以非事务运行 |
| NOT_SUPPORTED | 以非事务运行，有则挂起 |
| MANDATORY | 必须有事务，否则抛异常 |
| NEVER | 必须没有事务，否则抛异常 |
| NESTED | 有事务则嵌套（Savepoint），无则新建 |

**REQUIRES_NEW vs NESTED 区别**（面试高频追问）：
- REQUIRES_NEW：两个完全独立的事务，内层回滚不影响外层
- NESTED：通过 Savepoint 实现，外层可以回滚到 Savepoint，不是真正独立的事务

👤 **候选人回答**：说出默认开启、已有事务则加入，结合下单场景（ABC 三个表同事务）举例

📊 **评价**：✅ 核心思路正确，结合实际项目举例。可补充完整七种列表和 REQUIRES_NEW vs NESTED 的区别

💡 **面试官补充讲解**：补充了七种传播机制完整表格和 REQUIRES_NEW vs NESTED 关键区别

### 1.3 中间件与数据存储

**Q6: MySQL MVCC 机制？RR 和 RC 的区别？**

📝 **参考答案（完整版）**：

**MVCC 三大组件**：隐藏字段（DB_TRX_ID、DB_ROLL_PTR）+ undo log 版本链 + ReadView

**ReadView 判断逻辑**：
```
undo log 版本链上每个 trx_id：
  = creator_trx_id     → ✅ 可见（自己的修改）
  < min_trx_id         → ✅ 可见（已提交的旧事务）
  ≥ max_trx_id         → ❌ 不可见（未来的事务）
  在 m_ids 中          → ❌ 不可见（活跃事务）
  不在 m_ids 中        → ✅ 可见（已提交的事务）
```

**RR vs RC 核心区别**：
- **可重复读（RR）**：一个事务中**只创建一次 ReadView**（第一次 SELECT 时），后续复用。所以同一事务多次读结果一致
- **读已提交（RC）**：**每次 SELECT 都创建一个新 ReadView**。两次 SELECT 间有其他事务提交，第二次就能读到

**记忆口诀**：RR 是"拍一次照片看全程"，RC 是"每次拍照看当前"

👤 **候选人回答**：知道隐藏字段、undo log、ReadView 三大组件；RR 读同一次视图，RC 根据最近活跃事务读取

📊 **评价**：⚠️ 方向正确，知道三大组件和 RR/RC 的 ReadView 差异，但表述不够结构化

💡 **面试官补充讲解**：系统梳理了 ReadView 的五条判断规则，补充了 RR 和 RC 的 ReadView 创建时机差异

**Q7: Redis 大 Key 的影响、发现和处理？**

📝 **参考答案（完整版）**：

**大 Key 的影响**：
1. **阻塞请求**：Redis 单线程，`HGETALL` 百万字段 Hash 长时间占用 CPU
2. **网络带宽打满**：大 Key 返回数据量大
3. **集群倾斜**：Redis Cluster 中该分片成为热点
4. **持久化影响**：RDB/AOF 重写时 COW 内存飙升

**发现方式**：
- `redis-cli --bigkeys`：遍历扫描各种数据结构最大 key
- `MEMORY USAGE key`：查看单个 key 内存
- `SLOWLOG GET`：慢查询日志
- `MONITOR`：实时监控（慎用生产）

**处理方法**：
1. **拆分**：大 Hash 拆小 Hash，按业务分片
2. **压缩**：value 用 Snappy/Zstd 压缩
3. **本地缓存**：热点大 Key 在 JVM 本地缓存
4. **数据结构优化**：Hash 字段多时考虑拆 String

👤 **候选人回答**：知道影响查询速率；发现方式大致知道但忘了具体命令；处理是改业务逻辑杜绝大 Key

📊 **评价**：⚠️ 基本概念有，但发现方式（`--bigkeys`）和处理方案（拆分/压缩/本地缓存）细节不全

💡 **面试官补充讲解**：系统补充了四个影响维度、四种发现方式和四种处理方案

### 1.6 AI / Agent

**Q11: MCP vs Function Calling 的区别？**

📝 **参考答案（完整版）**：

**概念纠正**：Function Calling 也是大模型自主决定调用的，不是用户手动调。它是模型输出结构化 JSON 表示"我想调这个函数"，由应用层执行。

**本质区别**：

| 维度 | Function Calling | MCP |
|------|----------------|-----|
| **层次** | 模型能力层（JSON 格式约定） | 系统工程层（服务架构） |
| **功能** | 模型输出 JSON 格式的函数调用 | 定义工具接入体系（Tool/Resource/Prompt） |
| **状态** | 单次调用，无状态 | 会话级，上下文管理 |
| **连接** | 内存中定义 | 远程/本地 Server + JSON-RPC 2.0 |
| **生态** | 模型厂商各自定义 | 跨模型/跨平台统一标准 |

**一句话**：Function Calling 是"模型表达调用意图的格式"，MCP 是"如何组织、注册、调用工具的完整协议"，两者是上下游互补关系。

👤 **候选人回答**：MCP 有状态、Function Calling 无状态；MCP 大模型决定调用、Function Calling 用户手动调用（概念有部分偏差）；互补关系判断正确

📊 **评价**：⚠️ 互补关系判断正确。但"Function Calling 是用户手动调用"表述有偏差，实际上 FC 也是模型自主决定的。整体认知有进步

💡 **面试官补充讲解**：纠正了 Function Calling 的认知偏差，补充了五维度对比表和"上下游互补关系"的总结

### 1.7 算法与编程

**Q13: LeetCode 3. 无重复字符的最长子串（滑动窗口 + 哈希表）**

📝 **参考答案（完整版）**：

**思路**：滑动窗口 + HashMap 记录字符上次出现位置

```java
class Solution {
    public int lengthOfLongestSubstring(String s) {
        int ans = 0;
        HashMap<Character, Integer> hash = new HashMap<>();
        int left = 0; 
        for(int i = 0; i < s.length(); i++) {
            if(hash.containsKey(s.charAt(i))) {
                left = Math.max(left, hash.get(s.charAt(i)) + 1);
            }
            hash.put(s.charAt(i), i);
            ans = Math.max(ans, i - left + 1);
        }
        return ans;
    }
}
```

**复杂度**：时间 O(n)，空间 O(min(m, n)) 其中 m 为字符集大小

**关键设计点**：
- `left = Math.max(left, hash.get(s.charAt(i)) + 1)` 处理字符出现在 left 左边界的边界情况
- 先更新 left，再 put 当前字符，顺序不能反

👤 **候选人回答**：思路清晰，代码完全正确无 bug

📊 **评价**：✅ 代码干净，逻辑正确，相比前两次（LRU 无法手写、桶排序非标准解）显著进步
