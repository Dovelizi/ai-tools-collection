### 🆕 8. MySQL ICP vs 覆盖索引精确区分

- **领域/状态**：MySQL | 🆕 04-20 新暴露，❌ 概念错误
- **暴露**：1 次（04-20）
- **缺失点**：
  - [04-20] 把"status 在索引里的正常过滤"当成 ICP（错误）
  - [04-20] 覆盖索引思路偏（以为 ID 作主键就行）
- **学习方法**：
  1. **核心定义**：
     - **ICP** = **索引外的条件下推到索引层判断**，减少回表（EXPLAIN: `Using index condition`）
     - **覆盖索引** = SELECT 列都在辅助索引里，完全不回表（EXPLAIN: `Using index`）
  2. **经典例子**：
     ```sql
     INDEX idx_name_age(name, age)
     SELECT * FROM user WHERE name LIKE '张%' AND age = 20;
     -- 没 ICP：索引找 name → 全部回表 → 再过滤 age
     -- 有 ICP：索引层一起过滤 age → 只回表符合的
     ```
  3. **覆盖索引建法**：SELECT 列 + WHERE 列都放进联合索引
- **推荐资源**：
  - 📖 [一文看懂MySQL索引下推 ICP](https://blog.csdn.net/weixin_46425661/article/details/143624966)
  - 📖 [MySQL 2026 版索引全解](https://cloud.tencent.com/developer/article/2654428)
  - 📖 [JavaGuide MySQL 索引详解](https://javaguide.cn/database/mysql/mysql-index.html)
- **时长**：1 小时
- **验证**：能用 `name LIKE + age` 经典例子讲清两者区别

### 🆕 9. LinkedHashMap 实现 LRU + LinkedList.remove O(N) 坑（从 P1 升级）

- **领域/状态**：Java 集合 / 算法 | 🔄 连续 2 次未掌握（04-17、04-20）
- **暴露**：2 次
- **缺失点**：
  - [04-17] 思路对但拒绝写代码
  - [04-20] 愿意写了 📈，但用 `Deque<Integer>+HashMap<K,V>` 踩了 `LinkedList.remove(Object) O(N)` 坑
- **学习方法**：
  1. **背 LinkedList.remove(Object) 是 O(N) 的原因**：需遍历链表找节点（HashMap 只是判存在，不能定位节点）
  2. **方式 A（HashMap+自定义双向链表）3 技巧**：
     - HashMap 的 value 存 **Node 引用**（关键）→ O(1) 定位节点
     - Node 必须存 key → 淘汰时从 HashMap remove(key)
     - 哨兵头尾节点 → 免 null 判断
  3. **方式 B（LinkedHashMap）2 点**：
     - `super(capacity, 0.75f, true)` → `accessOrder=true`
     - 重写 `removeEldestEntry()` → 超容量自动淘汰
  4. 闭卷手写两种方式
- **时长**：1.5 小时
- **验证**：闭卷手写方式 A + B，能解释"为什么 Deque+HashMap 不是 O(1)"

### 🆕 11. CompletableFuture 三大要点（从 P1 升级）

- **领域/状态**：Java 并发 | 🔄 **连续 3 次缺失**（04-12、04-17、04-20），❌ 代码模板未掌握
- **暴露**：3 次
- **缺失点**：
  - [04-12] BFF 并行编排未提
  - [04-17] 提了但没说三大要点
  - [04-20] 默认线程池说对但隐患模糊；超时+降级组合"没用过"；thenApply/Async 说成"加快速度"
- **学习方法**：
  1. **背代码模板（BFF 并行降级）**：
     ```java
     ExecutorService executor = ...;
     CompletableFuture<ResultA> futureA = CompletableFuture
         .supplyAsync(() -> serviceA.call(), executor)   // ① 自定义线程池
         .orTimeout(2, TimeUnit.SECONDS)                  // ② 单独超时
         .exceptionally(e -> ResultA.fallback());         // ③ 单独降级
     CompletableFuture.allOf(futureA, futureB, ...).join();
     ```
  2. **ForkJoinPool.commonPool() 三大隐患**：
     - 线程数太少（CPU 核数-1）
     - 全局共享污染（一个慢任务拖垮所有）
     - 阻塞任务灾难（CPU 密集设计遇 IO 任务卡死）
  3. **thenApply vs Async 核心**：线程切换控制，不是速度
  4. **⚠️ 坑**：不要在 `allOf().get(timeout)` 设超时（整体超时），必须挂在每个 Future 上
  5. **实操**：写一个 BFF 8 并行降级 Demo
- **推荐资源**：
  - 📖 [CompletableFuture 异步编程全解](https://cloud.tencent.com/developer/article/2653680)
  - 📖 [CompletableFuture 自定义线程池实战](https://www.cnblogs.com/wikiman/p/17995877)
  - 📖 [exceptionally/handle/whenComplete 对比](https://blog.csdn.net/weixin_42531925/article/details/160298199)
- **时长**：2 小时
- **验证**：白板手写 3 行核心模板 + 写 BFF Demo

### 🆕 19. 偏向锁禁用 + Mark Word + JOL 工具

### 🆕 20. InnoDB 辅助索引存主键设计

- **领域/状态**：MySQL | 🆕 04-20 新暴露
- **学习方法**：
  1. **核心原因**：应对数据行的"移动"——页分裂/变长字段更新导致行移动，存主键可避免更新所有辅助索引
  2. **对比 MyISAM**：MyISAM 存物理地址（行移动要改所有索引） vs InnoDB 存主键（免维护成本）
  3. **为什么推荐自增主键**：单调递增 → 追加到 B+ 树最右 → 很少页分裂
- **时长**：0.5 小时
