# Java GC 面试知识全景图

## 一、如何判断对象是垃圾？

### 1. 引用计数法（了解即可）
- 每个对象维护一个引用计数器，被引用 +1，引用失效 -1，计数为 0 就是垃圾
- **致命缺陷：无法解决循环引用**（A 引用 B，B 引用 A，计数永远不为 0）
- Java **不使用**此方法，Python 使用

### 2. 可达性分析（Java 采用）
- 从 **GC Roots** 出发，沿引用链遍历，不可达的对象就是垃圾
- **GC Roots 包括**（面试必背）：
  - 虚拟机栈中引用的对象（局部变量）
  - 方法区中类静态属性引用的对象（`static` 变量）
  - 方法区中常量引用的对象（`final static`）
  - 本地方法栈中 JNI 引用的对象

> **面试口诀**：栈引用、静态属性、常量、JNI — 四种 GC Root

## 二、三种基础垃圾回收算法

| 算法 | 原理 | 优点 | 缺点 | 适用代 |
|------|------|------|------|--------|
| **标记-清除** | 先标记存活对象，然后清除未标记的 | 简单，不需要移动对象 | **内存碎片**，分配大对象时可能找不到连续空间 | 老年代（CMS） |
| **标记-整理** | 先标记存活对象，然后全部向一端移动，清理边界外内存 | **无碎片** | 移动对象开销大，需要 STW | 老年代（Serial Old） |
| **复制算法** | 内存一分为二，只用其中一半，GC 时把存活对象复制到另一半 | **无碎片**，速度快 | 浪费一半内存空间 | **新生代**（Eden + S0/S1） |

> **新生代为什么用复制算法？** 因为新生代对象朝生夕灭，存活率极低（约 2%），复制的对象少，效率极高。HotSpot 把新生代分成 **Eden : S0 : S1 = 8:1:1**，每次只浪费 10% 空间。

## 三、JVM 内存分代模型

```
堆内存
├── 新生代（Young Generation）—— 复制算法
│   ├── Eden 区（80%）：新对象优先在此分配
│   ├── Survivor 0（10%）：From 区
│   └── Survivor 1（10%）：To 区
│
├── 老年代（Old Generation）—— 标记-清除 或 标记-整理
│   └── 长期存活的对象（经过 15 次 Minor GC 仍存活）
│
└── （JDK 8+）元空间（Metaspace）—— 本地内存，存类信息
```

**对象晋升到老年代的条件**（面试常问）：
1. 年龄达到阈值（默认 15，`-XX:MaxTenuringThreshold`）
2. 大对象直接进入老年代（`-XX:PretenureSizeThreshold`）
3. Survivor 区中相同年龄对象总大小超过 Survivor 的一半，≥ 该年龄的对象直接晋升
4. Minor GC 后 Survivor 放不下，直接进入老年代

---

## 四、GC 类型

| GC 类型 | 回收范围 | 触发条件 |
|---------|---------|---------|
| **Minor GC** | 新生代 | Eden 区满 |
| **Major GC** | 老年代 | 老年代空间不足 |
| **Full GC** | 整个堆 + 方法区 | 老年代不足、方法区不足、`System.gc()`、空间分配担保失败 |

## 五、七种垃圾回收器详解

### 总览表（面试必背）

| 收集器 | 作用代 | 算法 | 线程 | STW | 特点 | JDK 状态 |
|--------|--------|------|------|-----|------|----------|
| **Serial** | 新生代 | 复制 | 单线程 | 长 | 简单高效，客户端默认 | 可用 |
| **ParNew** | 新生代 | 复制 | 多线程 | 中 | Serial 多线程版，配合 CMS | 可用 |
| **Parallel Scavenge** | 新生代 | 复制 | 多线程 | 中 | 关注**吞吐量** | JDK 8 默认 |
| **Serial Old** | 老年代 | 标记-整理 | 单线程 | 长 | CMS 备用方案 | 可用 |
| **Parallel Old** | 老年代 | 标记-整理 | 多线程 | 中 | 配合 Parallel Scavenge | JDK 8 默认 |
| **CMS** | 老年代 | 标记-清除 | 并发 | 短 | 关注**低延迟** | JDK 14 移除 |
| **G1** | 全堆 | 标记-整理/复制 | 并发 | **可控** | Region 分区，可预测停顿 | **JDK 9+ 默认** |
| **ZGC** | 全堆 | 标记-整理 | 并发 | **亚毫秒** | 着色指针+读屏障 | JDK 15+ 生产可用 |

### 5.1 CMS（Concurrent Mark Sweep）— 了解即可

**目标**：最短停顿时间（低延迟）

**四阶段**：
1. **初始标记（STW）**：标记 GC Roots 直接引用的对象，极快
2. **并发标记**：与用户线程并发，遍历对象图
3. **重新标记（STW）**：修正并发标记期间变动的引用（增量更新算法）
4. **并发清除**：与用户线程并发，清除垃圾对象

**致命缺点**（面试必问）：
- **内存碎片**：标记-清除不整理，导致碎片堆积，最终触发 Full GC（退化为 Serial Old，长时间 STW）
- **浮动垃圾**：并发清除期间新产生的垃圾只能等下次 GC
- **CPU 敏感**：并发阶段占用 CPU 资源（默认回收线程 = CPU 核数 / 4）

> **记忆口诀**：CMS 四阶段 — 初标（STW）、并标、重标（STW）、并清。缺点三连 — 碎片、浮动垃圾、吃 CPU。

---

### 5.2 G1（Garbage-First）— 重点掌握

**目标**：可预测的停顿时间 + 无碎片

**核心设计**：
- 把堆划分为 **约 2048 个大小相等的 Region**（1-32MB）
- 每个 Region 动态扮演 **Eden / Survivor / Old / Humongous**（大对象，≥ Region 50%）
- 不再物理分代，而是**逻辑分代**

**G1 工作流程**：

```
Young GC（新生代回收）：
  Eden 满 → 存活对象复制到 Survivor Region → 年龄到了晋升到 Old Region

Mixed GC（混合回收，G1 特有）：
  当老年代占比达阈值（-XX:InitiatingHeapOccupancyPercent，默认 45%）
  → 触发并发标记 → 然后同时回收新生代 + 部分老年代（选回收价值最高的 Region）

Full GC（兜底，尽量避免）：
  Mixed GC 来不及回收 → 退化为单线程 Full GC（要极力避免）
```

**并发标记四阶段**：
1. **初始标记（STW）**：标记 GC Roots 直接引用，搭 Young GC 的便车执行
2. **并发标记**：与用户线程并发，用 **SATB（Snapshot-At-The-Beginning）** 算法追踪引用变化
3. **最终标记（STW）**：处理 SATB 队列中残留的引用
4. **筛选回收（STW）**：按回收价值排序 Region，在 **目标停顿时间内** 选择性回收

**关键特性 — 可预测停顿**：
- `-XX:MaxGCPauseMillis=200`（默认 200ms）
- G1 根据每个 Region 的 **回收成本和垃圾量** 动态选择回收哪些 Region
- 目标时间内回收收益最大的 Region → 这就是 "Garbage-First" 名字的由来

**G1 vs CMS 对比**（面试必背）：

| 维度 | CMS | G1 |
|------|-----|-----|
| 内存布局 | 固定分代（新生代/老年代） | **Region 动态分代** |
| 回收算法 | 标记-清除（有碎片） | **复制/标记-整理（无碎片）** |
| 停顿控制 | 不可预测 | **可设目标停顿时间** |
| 碎片问题 | 严重（最终 Full GC） | **无碎片（Region 间复制）** |
| 适用堆大小 | 堆 < 8G | **堆 ≥ 4G（越大优势越明显）** |
| JDK 状态 | JDK 14 移除 | **JDK 9+ 默认** |

**G1 关键参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:MaxGCPauseMillis` | 200ms | 目标停顿时间 |
| `-XX:G1HeapRegionSize` | 自动计算 | Region 大小（1-32MB，2 的幂） |
| `-XX:InitiatingHeapOccupancyPercent` | 45% | 老年代占堆比例达多少触发并发标记 |
| `-XX:G1MixedGCCountTarget` | 8 | Mixed GC 次数目标 |

> **面试口诀**：G1 三板斧 — **Region 分区、优先回收、可预测停顿**。

### 5.3 ZGC（Z Garbage Collector）— 了解原理

**目标**：**亚毫秒级停顿**（< 1ms），不随堆大小增长

**核心技术**：

**1. 着色指针（Colored Pointers）**：
- 在 64 位指针中，利用高 4 位存储 GC 元数据：
  ```
  [64位地址] = [未使用 16位] + [4位 GC 标记] + [44位 实际地址]
  
  4位标记：
  - Finalizable（是否要执行 finalize）
  - Remapped（是否已重映射到新地址）
  - Marked0 / Marked1（标记位，交替使用）
  ```
- **好处**：GC 信息直接编码在指针里，无需额外的标记位图，效率极高

**2. 读屏障（Load Barrier）**：
- 每次从堆中**读取对象引用**时触发（注意：是读屏障，不是写屏障）
- 读屏障检查指针的着色位，如果发现对象已经被移动，**自动修正指针**指向新地址
- 这使得对象移动可以和用户线程并发执行，**不需要 STW**

**3. 内存多重映射**：
- 同一块物理内存映射到 3 个虚拟地址（Marked0/Marked1/Remapped）
- 通过切换虚拟地址的映射实现并发标记和重定位

**ZGC 工作流程**：
1. **并发标记**：从 GC Roots 遍历对象图，通过着色指针标记存活对象
2. **并发预备重分配**：选择需要回收的 Region 集合
3. **并发重分配**：将存活对象复制到新 Region，旧地址通过**转发表**指向新地址
4. **并发重映射**：修正所有指向旧地址的引用（读屏障也会自愈修正）

**ZGC 的 STW 极短的原因**：
- 只有初始标记和初始重映射需要极短的 STW（仅扫描 GC Roots，微秒级）
- 标记、整理、指针修正全部**并发**完成
- 不管堆多大（1G 还是 16TB），停顿时间都 < 1ms

**G1 vs ZGC 对比**：

| 维度 | G1 | ZGC |
|------|-----|-----|
| 停顿时间 | 可控（通常 100-200ms） | **亚毫秒（< 1ms）** |
| 最大堆 | 一般 < 64G | **最大 16TB** |
| 核心技术 | SATB + Region 复制 | **着色指针 + 读屏障** |
| 分代 | 逻辑分代 | JDK 21 引入分代 ZGC |
| 适用场景 | 通用（4G-64G 堆） | **超低延迟（金融/交易系统）** |
| JDK 状态 | JDK 9+ 默认 | JDK 15+ 生产可用，JDK 21 分代 ZGC |

## 六、面试高频追问 Q&A

### Q1：你们线上用什么 GC？为什么选它？

> 线上用 G1。我们的服务堆大小 8G，日均百万级请求，选 G1 的原因：
> 1. 可预测停顿（设 MaxGCPauseMillis=200ms），避免长时间 STW 影响用户体验
> 2. 无碎片问题，不像 CMS 最终可能退化为 Full GC
> 3. JDK 9+ 默认收集器，社区支持和优化最积极

### Q2：G1 什么时候会退化为 Full GC？怎么避免？

> **触发 Full GC 的场景**：
> 1. 并发标记来不及完成，老年代被填满（**Concurrent Mode Failure**）
> 2. 晋升失败：Young GC 时 Survivor 和 Old 都放不下存活对象
> 3. 大对象分配失败：找不到连续的 Humongous Region
>
> **避免方式**：
> - 适当增大堆内存
> - 降低 `InitiatingHeapOccupancyPercent`（提前触发并发标记）
> - 排查大对象分配和内存泄漏

### Q3：SATB 和增量更新的区别？（G1 vs CMS 的标记算法）

> 两者都是解决**并发标记期间引用变化**的问题：
> - **CMS 用增量更新**：记录新增的引用关系，重新标记时重新扫描这些新引用
> - **G1 用 SATB（原始快照）**：在标记开始时对引用关系拍快照，标记期间删除的引用也不放过（宁可多标不漏标）
>
> SATB 的好处是最终标记阶段更快（只需处理 SATB 队列），缺点是可能有少量浮动垃圾。

### Q4：什么情况下考虑用 ZGC？

> 对延迟极度敏感的场景：金融交易系统、实时竞价广告、在线游戏服务器。
> 或者超大堆（> 64G）的场景，G1 的停顿时间会随堆增大而增加，ZGC 则不会。

### Q5：你做过 GC 调优吗？说说思路。

> **GC 调优三步法**：
> 1. **看日志**：开启 GC 日志（`-Xlog:gc*`），观察 GC 频率、停顿时间、各代大小
> 2. **定目标**：明确优化目标（低延迟选 G1/ZGC、高吞吐选 Parallel）
> 3. **调参数**：
>    - 停顿长 → 调 `MaxGCPauseMillis` 或增大堆
>    - Young GC 频繁 → 增大新生代（`-Xmn` 或调 `G1NewSizePercent`）
>    - Full GC → 排查内存泄漏，降低 `InitiatingHeapOccupancyPercent`
>
> 结合项目说："线上出现过 Young GC 频繁的问题，通过 GC 日志发现是 Eden 太小，调大后 GC 频率降了一半。"

## 七、速记卡片

```
判断垃圾：可达性分析（4 种 GC Root：栈引用/静态属性/常量/JNI）

三种算法：
  复制 → 新生代（快，浪费空间）
  标记-清除 → CMS（碎片）
  标记-整理 → G1/ZGC（无碎片，移动对象）

收集器演进：
  CMS（低延迟但有碎片，JDK14 移除）
  → G1（Region + 可预测停顿，JDK9 默认）
  → ZGC（着色指针 + 读屏障，亚毫秒停顿）

G1 三板斧：Region 分区、优先回收、可预测停顿
ZGC 两板斧：着色指针、读屏障

CMS 四阶段：初标(STW) → 并标 → 重标(STW) → 并清
G1  四阶段：初标(STW) → 并标 → 终标(STW) → 筛选回收(STW)
ZGC 全并发：只有 GC Root 扫描需极短 STW（微秒级）
```

> 建议先背 **G1（面试出现率 90%）**，再了解 **CMS（对比用）** 和 **ZGC（加分项）**。
